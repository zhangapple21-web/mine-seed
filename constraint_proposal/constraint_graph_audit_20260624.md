## 一、提案内容

在M04 Constraint提案模块中增加**约束图谱审计**维度，从"约束是否存活"扩展到"约束之间如何相互影响"。

### 当前审计维度（已有）

| 维度 | 指标 | 说明 |
|------|------|------|
| 存活率 | 被调用次数/总运行次数 | 约束是否在使用 |
| 半衰期 | 从生效到首次被违反的时间 | 约束的衰减速度 |
| 覆盖率 | 被约束保护的场景数 | 约束的保护范围 |

### 新增审计维度（提案）

| 维度 | 指标 | 说明 |
|------|------|------|
| **图谱密度** | 约束间影响关系数/总可能关系数 | 约束之间是否高度耦合 |
| **传递冲突** | A约束→影响决策→B约束被违反的链路数 | 隐式冲突 |
| **孤岛率** | 不与任何其他约束产生影响的约束占比 | 约束是否有全局作用 |
| **Winchester风险** | 语义上互相矛盾的约束对数 | 约束之间是否打架 |

## 二、当前约束图谱分析

### 2.1 约束清单

基于生产域的19条约束 + 研究域的7条活跃原则：

| ID | 类型 | 内容 | 图谱角色 |
|----|------|------|---------|
| C000 | 铁律 | 不要为了填补空白而建设 | **锚点**（L∞层） |
| A001 | 原则 | 观察先于行动 | C000的具体化 |
| A002 | 原则 | Reality≠Observation | A001的深化 |
| A003 | 原则 | 约束必要性>数量 | **与C000直接关联** |

### 2.2 已识别的影响关系

```
C000（铁律）──→ A003（约束必要性>数量）
  │                  "不要建设" → "少加约束"
  │
  ├──→ A001（观察先于行动）
  │        "先确认空白是否生长" → "先怀疑地图"
  │
  └──→ A007（反思太快是急躁）
           "不要急着建设" → "不要急着反思"

A001（观察先于行动）──→ A002（Reality≠Observation）
  │                         "先观察" → "观察可能不真实"
  │
  └──→ A004（反思触发以重要性积累为主）
           "观察积累够了再行动" → "积累够了再反思"

A004 ──┐
       ├──→ 互补：A004是"什么时候反思"，A007是"反思不要太快"
A007 ──┘
```

### 2.3 潜在冲突识别

| 冲突对 | 描述 | 严重度 |
|---------|------|--------|
| A004 vs A007 | A004说"积累够了就反思"，A007说"反思太快是急躁" | 低（互补，非矛盾） |
| A003 vs 新约束 | "少加约束"可能阻止必要的约束 | 中（需要逐案判断） |
| #033 vs M01实战 | #033"成交量辅助波动率"与M01"波动率驱动成交量"方向相反 | **高**（因果倒置） |

### 2.4 Catacombs风险

**xz后门类比**：C000作为铁律（最深层约束），如果被"绕过"（如Agent觉得某个建设是"必要的"而非"填补空白的"），可能导致系统性偏移。

**Winchester Mystery House风险**：知识班每天产出1条Constraint提案（#033），如果这些提案未经图谱审计就直接入库，约束系统会变成迷宫。

## 三、实施建议

1. **约束图谱可视化**：每条约束作为节点，影响关系作为边，绘制有向图
2. **传递冲突检测**：新增约束时，检查是否会通过影响链间接违反已有约束
3. **Winchester审计**：每月检查约束之间是否存在语义矛盾
4. **图谱密度控制**：密度超过阈值时，触发约束精简流程
5. **孤岛约束清理**：不与任何其他约束产生影响的约束，评估是否需要保留

---

## 四、局限性

1. 当前分析基于有限数据（7条活跃原则 + 19条生产约束），图谱规模较小
2. 影响关系的判定带有主观性，不同人可能对约束间关系有不同理解
3. "传递冲突"的检测需要更形式化的方法，当前只能靠人工分析
4. 没有生产域约束的完整文本，无法做全量分析
