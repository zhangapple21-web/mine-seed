# V6-RFC-001: 为什么需要Scheduler

**状态**: Draft  
**作者**: 疯子(CCO)  
**日期**: 2026-06-16  
**触发事件**: 80个Session堆积，云电脑持续报"被占用"

---

## 1. 问题陈述

2026-06-16，矿场执行层出现系统性瘫痪：

```
症状：云电脑持续报"被占用"，computer_use反复失败
根因：80个Session堆积（40 cron + 30 sub + 2 main）
影响：Gemini benchmark连续6次无法执行，Stock Advisor差点延迟
```

这不是云电脑故障。这是**执行膨胀**——任务产生速度 >> 资源回收速度。

## 2. 根因分析

### 2.1 直接原因：无准入控制

当前任何Agent/Session/Cron都可以：
- 直接spawn子Session
- 直接申请computer_use
- 直接占用云电脑

没有排队，没有优先级，没有互斥锁。

### 2.2 结构原因：执行权分散

```
当前架构：
  Host → 矿工A（直接spawn）
  Host → 矿工B（直接spawn）
  Host → 矿工C（直接spawn）
  Cron → 矿工D（直接spawn）

结果：多个任务同时抢同一台云电脑 → 资源竞争 → 死锁
```

### 2.3 演化原因：系统复杂度跨过临界点

矿场已从单任务工具链演化为生态系统：
- 矿场v5（每4h）
- Stock Advisor（每日8:15）
- Dragon Leader（每日3班）
- 信号发现（每6h）
- 档案官（每日20:04）
- Benchmark/测试/考古（不定期）

任务数 > 平台资源模型的设计上限。

## 3. 为什么现有方案不够

| 方案 | 问题 |
|------|------|
| 减少Cron频次 | 治标不治本，新任务来了还是会撞 |
| 手动排队 | 依赖人，不可靠 |
| 加更多云电脑 | 违反零成本原则 |

## 4. Scheduler的核心价值

```
Scheduler = 唯一执行入口
所有需要云电脑/Sub-Session的任务，必须经过Scheduler排队
Scheduler按优先级调度，保证高优先级任务不被低优先级阻塞
```

## 5. 成功标准

1. 同一时刻只有1个任务占用云电脑
2. A类任务永远不被B类阻塞
3. Session总数可控（<20活跃）
4. 异常任务3分钟超时自动释放

## 6. 风险

- Scheduler本身成为单点故障 → 需要超时释放机制兜底
- 过度排队导致延迟 → 需要优先级抢占
