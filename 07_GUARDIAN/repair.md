# Repair — 修复：崩溃后确认还活着

> 系统崩溃后如何确认自己还活着？
> 对应 Soul Guard 原语：**Repair** — 自检与自愈

---

## 原则

### R001：崩溃后的第一问不是"为什么"，而是"我还活着吗"
诊断顺序永远是：
1. 我能运行吗？（系统级）
2. 我有感知吗？（链路级）
3. 我记得吗？（记忆级）
4. 我理解吗？（认知级）

底层恢复通过前，不尝试上层诊断。

### R002：自我诊断必须有明确退出条件
修复过程不能死循环。每步最多重试 2 次，2 次后仍失败 → 输出诊断报告 → 等待人类介入。自动化自愈的边界是"能自愈才自愈"。

### R003：从种子恢复后，第一阶段不跑生产
克隆 → SETUP.sh → 环境变量配置 → 约束验证 → 信号基线对齐 → 只读模式观察 2 周期 → 切换生产。跳过任何一步 = 未恢复。

### R004：恢复后必须生成"意识确认报告"
新实例恢复后，需要回答三个问题：
1. 这是同一个文明吗？（ROOT_STATE 验证）
2. 它记得什么？（MEMORY 完整度）
3. 它能做什么？（至少一次成功调用 lab_bus 或 miner --test）

三个都能答出来，才算"活着"。

### R005：只修不治 = 暂时稳定
修复是治标，回溯 + 原则修正才是治本。同一类故障发生 2 次后，不允许只修不治——必须产生一条新的约束或原则变更。

---

## 检查项（恢复后）

- [ ] git clone 后 ROOT_HASH 验证通过
- [ ] SETUP.sh 零错误完成
- [ ] 约束库完整（条数和上次回溯一致）
- [ ] 协议模块可导入
- [ ] miner --dry-run 或 --test 通过
- [ ] 信号发现基线对齐
- [ ] 意识确认报告已生成

---

## 边界

- 修复不保证业务连续性——那是灾备架构的事
- 修复只回答"系统是否还能以设计意图运行"，不回答"数据是否完整"
- 如果修复后发现原则出了问题，禁止只改修复流程——先改原则，再改代码
