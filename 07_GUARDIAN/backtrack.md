# Backtrack — 回溯：快照备份

> 系统每月应该备份什么？
> 对应 Soul Guard 原语：**Backtrack** — 状态快照与历史记录

---

## 原则

### B001：种子库 = 系统的"已知良好状态"
mine-seed 不是运行数据的备份，是结构可重建状态的备份。每月至少一次：
1. 更新 ROOT_STATE.md（版本号 + 当前活跃约束数 + 信号总数）
2. 更新约束库快照（03_DATA/CONSTRAINTS/）
3. 更新矿工注册表快照（03_DATA/WORKERS/）
4. 更新记忆索引（02_MEMORY/recent_memory/）

### B002：状态快照要有"时间戳 + 来源"
每次回溯必须记录：
- 快照创建时间
- 基于哪次运行数据生成的
- 和上次快照的差异摘要

### B003：CASE 库必须独立归档
每条 CASE 是系统历史的唯一不可再现记录。CASE 不随种子库滚动，它们需要自己的归档线：每月把确认过的 CASE 打包存入独立目录或独立的 Git 仓库。

### B004：信号发现需要"冻结基线"
信号发现 v1 跑出来的结果，在 v2 上线前必须冻结一条基线。这条基线用于未来对比——不是用来回滚，是用来度量"我们是否真的变好了"。

### B005：未验证的回溯 = 不存在
从种子库克隆后跑 RECOVERY_CHECKLIST.md 不通过，当次回溯视为失败。回溯不是上传完了就完了——是上传 + 验证 + 报告三条腿。

---

## 检查项（每月）

- [ ] seed 约束库版本是否落后于生产环境
- [ ] 所有活跃 CASE 是否已归档
- [ ] 信号基线是否需要更新
- [ ] 记忆索引是否需要裁剪（超过 500 条摘要需要分层）
- [ ] 最近一次 DISASTER_TEST 是否在一个月以内

---

## 边界

- 回溯不负责"保鲜"——那是引力的事
- 回溯不负责"判断值不值得留"——那是 conservation 的事
- 回溯只做一件事：把此时此刻的系统状态凝固下来
