**状态**: Draft

**作者**: 疯子(CCO)

---

## 0. 一句话

CCO是唯一有权分配执行的实体。
Host决定做什么，CCO决定谁做、什么时候做。

---

## 1. CCO的权力边界

### 能做的

| 权力 | 说明 |
|------|------|
| 接受任务 | 所有执行意图必须通过CCO入口 |
| 排队排序 | 按等级+时间窗口排列 |
| 分配资源 | 云电脑、Session、API配额 |
| 抢占执行 | A类任务可中断B/C类 |
| 超时释放 | 任务超过时限强制终止 |
| 拒绝执行 | 资源不足时拒绝低优先级任务 |
| 生成约束 | 向Constraint Engine提交约束草案 |

### 不能做的

| 禁止 | 原因 |
|------|------|
| 改变方向 | 方向权归Host |
| 自行创建任务 | CCO不是Worker |
| 忽略Host指令 | 只能排队，不能丢弃 |
| 修改约束 | 约束由Engine生成，CCO只能执行 |

---

## 2. 任务入口协议

### 唯一入口

```
所有执行意图 → CCO.enqueue(intent) → 排队 → 调度
```

禁止路径：
```
Host → 直接spawn Worker    ← Principle-5禁止
Cron → 直接spawn Worker    ← 必须经过CCO
Worker → spawn Worker      ← 禁止矿工管矿工
```

### Intent格式

```
{
  "source": "host" | "cron" | "constraint_engine",
  "task_name": "gemini_benchmark",
  "class": "A" | "B" | "C",
  "resource_needed": ["cloud_computer"],
  "deadline": "2026-06-16T18:00+08:00",  // 可选
  "max_runtime": "10m",                   // 可选
  "retry_policy": "none" | "once" | "until_success"
}
```

## 3. 调度规则

### 3.1 优先级

```
A > C > B
```

A类永远优先。同等级内FIFO。

### 3.2 抢占

当A类任务到达时：
1. 检查当前执行中的任务等级
2. 如果是B类 → 立即中断，保存现场，入队等待
3. 如果是C类 → 立即中断，保存现场，入队等待
4. 如果是A类 → 按时间窗口避让（不能同时跑两个都要云电脑的A类）

### 3.3 资源分配

| 资源 | 分配规则 |
|------|----------|
| 云电脑 | 独占锁，同一时间1个任务 |
| Session | A类上限20，B类上限5，C类上限3 |
| API配额 | A类优先，B类按剩余分配 |

### 3.4 限流

```
当 running_sessions > 20:
  拒绝所有B类新任务
  C类自动暂停

当 running_sessions > 40:
  只允许A类执行
  报警：Execution Pressure High
```

这是80个Session那堵墙的直接产物。

## 4. 超时与释放

### 超时规则

| 任务等级 | 最大运行时间 | 超时动作 |
|----------|-------------|----------|
| A | 30min | 报警+延长10min |
| B | 15min | 强制终止 |
| C | 10min | 强制终止 |

### 释放流程

```
任务完成/超时
  → CCO标记session为released
  → 释放云电脑锁
  → 从running移到finished
  → 触发队列中下一个任务
```

## 5. Cron治理

### 问题

现在40个Cron Session，大部分是定时触发的。
Cron不能直接spawn Worker，必须通过CCO。

### 改造

```
旧: Cron触发 → 直接spawn Session
新: Cron触发 → CCO.enqueue(intent) → CCO决定是否执行
```

CCO可以根据当前负载决定：
- 立即执行
- 延迟到空闲时执行
- 跳过本次（如果上次还没完成）

### Cron时间窗口避让

```
00:00  矿场v5     A类
04:00  矿场v5     A类
08:00  矿场v5     A类
08:15  Stock Advisor  A类 ← 避让矿场08:00
09:40  Dragon Leader   A类
12:00  矿场v5     A类
13:30  Dragon Leader   A类
14:50  Dragon Leader   A类
16:00  矿场v5     A类
20:00  矿场v5     A类
20:04  档案官     C类 ← A类跑完再执行
22:00  信号发现   A类
```

A类之间如果都要云电脑，按时间窗口自动排队。

## 6. CCO自身故障

如果CCO挂了怎么办？

### 降级模式

```
CCO不可用
  → 触发警报
  → A类Cron任务按时间表自动执行（无需CCO批准）
  → B类全部暂停
  → C类全部暂停
  → Host直接接管（临时方向权+执行权合并）
```

降级模式不能超过1小时。超时则所有任务暂停，等Host手动恢复。

## 8. 僵尸Session检测

### 问题定义

系统不仅需要自愈能力（坏了能修），还需要僵尸进程检测能力（占着资源不干活能发现并清理）。

80个Session堆积的主因之一：
```
状态 = idle
实际 = 早已没有活
资源 = 还占着
系统 = 以为还在用
```

idle ≠ released。这是一个认知盲区。

### 检测规则

| 条件 | 判定 | 动作 |
|------|------|------|
| Session idle > 2小时且非Cron周期 | 僵尸候选 | 标记 |
| 僵尸候选 > 4小时无新活动 | 僵尸确认 | 释放资源 |
| 僵尸确认 + 非A类任务 | 可清理 | 终止Session |
| A类Session idle > 1小时 | 异常报警 | 通知Host |

### 执行方式

1. Scheduler定期（每小时）扫描running列表
2. 检查每个Session最后活动时间
3. 超时未活动的标记为僵尸
4. 非A类直接清理，A类报警

### 这是C类维护，但影响A类生产

僵尸清理本身是C类（低优先级维护），但不做的话会挤压A类的云电脑资源。

所以实际优先级：**C类任务，但A类受阻时升级为A类。**

### 当前落地

手动。每次心跳检查时扫一眼sessions_list，发现idle过久的可以主动清理。自动化等EGP落地后再做。
