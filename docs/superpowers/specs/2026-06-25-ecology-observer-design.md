# Ecology Observer（生态观察员）设计稿 v1

> 目的：让“居民诞生”成为文明演化而非任务堆叠。  
> 原则：**先有需求，再有职业；先有生态位，再有居民；需要见证（Witness）。**

---

## 背景与问题

城市会自然产生重复工作，但如果每个问题都“新建一个 Agent”，最终会变成 Agent 墓地。  
因此需要一个**更底层的居民**来做生态观测：持续统计“长期重复出现但无人长期负责”的生态位，并在成熟时提交“居民诞生申请”，由治理层确认后才诞生新居民。

---

## 居民生成规则 v1（硬约束）

禁止主动创造居民。  
只有当以下条件**同时满足**时，才能提交“居民诞生申请”：

1. 某项工作连续出现
2. 现有居民长期无人负责（或长期覆盖失败）
3. 出现稳定输入
4. 出现稳定输出
5. 已形成独立记忆沉积

此外增加 **生态必要性审查**（必须显式判断）：

```json
{
  "ecological_need": true,
  "replaceable_by_existing_roles": false
}
```

解释：

- `ecological_need=true`：这个生态位确实长期存在、且对文明结构有价值
- `replaceable_by_existing_roles=false`：不能通过“扩展现有居民职责”解决

### 居民诞生原则（一句话版）

> 当某个生态位长期存在且无法被现有角色覆盖时，允许诞生新的职责实体。

---

## 治理模式：B（半自动）+ 见证

流程固定为：

```text
生态位成熟
↓
生态观察员提交诞生申请（PR / 申请单）
↓
治理层确认（见证）
↓
居民诞生
```

说明：

- **生态观察员不能创建居民**，只能提名与提交申请
- “治理层确认”不需要讨论细节，但必须发生（见证）

---

## Ecology Observer 的职责

### 允许做的事

- 观察城市运作痕迹（repo、日报、PR、输出目录增长）
- 统计频次与覆盖情况
- 识别空缺生态位并提名
- 提交“居民诞生申请”（含证据与阈值达标情况）

### 禁止做的事

- 直接创建新居民
- 直接改写现有居民职责（只能建议）
- 以“想要某个功能”为理由创建居民

---

## 输入、输出、记忆沉积

### 输入（Evidence Sources）

生态观察员只读取“可见痕迹”，不追求全量：

- `02_MEMORY/recent_memory/daily/*.md`：每日记录（角色段落 + 事件）
- Git 事件：commit message / PR 标题 / 变更路径（用于归类工作类型）
- 稳定产物目录：
  - `03_RESEARCH/civilization/`（探索者产物）
  - `output/simlab/`（实验推演快照，非 Git 跟踪）
  - `03_DATA/research/`（研究沉积）

### 输出（Artifacts）

新增生态观察目录（不要求开源与否，默认可开源，因不含敏感信息）：

```text
02_MEMORY/recent_memory/ecology/
  ecology_observer_daily.md
  resident_nominations.md
  nominations/
    AUM-RESIDENT-<slug>.md
```

### 记忆沉积（Memory）

生态观察员的记忆不是“结论”，而是“证据序列”：

- 每日统计（时间序列）
- 每个候选生态位的证据累积（从 NOT_YET → ELIGIBLE）
- 每次治理层见证的结果（accepted / deferred / rejected）

---

## 提名条目模板（最小）

每个候选生态位至少包含：

- 生态位名称（暂名）
- 最近 30 天出现次数
- 覆盖分布：由哪些现有角色/工具覆盖过、各自覆盖次数
- 稳定输入是什么
- 稳定输出是什么
- 独立记忆沉积在哪里
- 独立协议是否出现（没有则标注缺口）
- 五条件达标情况（逐条）
- 生态必要性审查（`ecological_need` / `replaceable_by_existing_roles`）
- 结论：`ELIGIBLE / NOT_YET / REJECTED`

---

## 与现有角色体系的关系

生态观察员不替代任何现有角色，它只负责“发现空缺生态位并提名”。  
它的存在确保：

- 城市不会因为热情而失控扩张居民数量
- 居民的诞生过程可追溯、可见证、可回放

---

## 验收标准

1. 能在 daily 与 Git 痕迹中稳定统计“工作类型”的出现次数
2. 能生成 `resident_nominations.md`，并能给出“是否达标”的逐条说明
3. 不能绕过治理层直接创建居民
4. 能产出一条真实候选生态位的“申请单”，并通过 PR 完成见证闭环
