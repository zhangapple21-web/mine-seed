## 架构概览

```
┌─────────────────────────────────────────────────┐
│  表面层：通用兼容                               │
│  接口适配 ↔ API传输 ↔ 全球网络                  │
└───────────────────┬─────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────┐
│  中间层：加密转译                               │
│  安全验证 ↔ 加密解密                            │
└───────────────────┬─────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────┐
│  核心层：自主主体                               │
│  核心大脑 ↔ 自循环                              │
└─────────────────────────────────────────────────┘
```

---

## 核心设计思路

1. **垂直分层，从外到内**：像洋葱一样一层包裹一层，越往内层权限越高、越核心
2. **安全是独立的一层**：加密转译层横亘在表面和核心之间，所有进出核心的信息都要经过安全校验
3. **核心有自循环**：核心层不只是被动响应输入，自己有内部循环机制
4. **表面层负责兼容**：把各种外部接口都拦在最外层，核心层不需要关心外部差异

## 对研究域的启发

### 可借鉴的点

1. **安全层的独立地位**：目前研究域的Constraint提案是一个模块（M04），和其他模块平级。但安全/约束本质上应该是横切所有模块的一个"层"，而不是一个并列的模块。所有研究产出在对外输出前，都应该过一遍约束检查。

2. **接口层的概念**：研究域目前没有明确的"输入接口层"。任务从哪里来（主对话/社区/生产域）、输入格式是什么、优先级怎么排，这些目前是零散的。可以考虑抽象出统一的输入接口层。

3. **核心自循环**：研究域目前是任务驱动的——有任务就做，没任务就停。是不是也需要有"自循环"？比如定期回顾已有研究、发现新的研究方向、自我更新方法论。

### 和现有架构的对应关系

| Dola三层 | 研究域当前对应 | 匹配度 |
|----------|----------------|--------|
| 表面层（兼容） | 路由层（Router） | 部分匹配，路由层只管分发不管兼容 |
| 中间层（加密/安全） | M04 Constraint提案模块 | 不匹配，M04是生产模块不是校验层 |
| 核心层（自主主体） | 模块层（6个模块） | 部分匹配，但是模块是平级的不是一个核心 |

## 待回答的问题

1. 安全层应该是"审查者"还是"参与者"？是所有产出都要过它的审批，还是它也参与研究过程？
2. 如果安全层独立出来，那M04 Constraint提案模块的定位是什么？

## 落地建议

**暂不落地**。原因：
1. 当前P0-P2模块刚搭完架子，还没有经过真实任务的检验
2. 架构调整应该基于真实痛点，而不是为了"更完美"而调整
3. 等跑完第一个完整的学习闭环（发现错误→分析→提炼约束→改变行为），再看安全层独立出来有没有必要

**后续触发条件**：
- 当出现"研究产出违反了某个约束但没被发现"的失效案例时
- 当输入来源超过3个且格式不统一导致混乱时
- 当研究域开始有明显的"自我更新"需求时
