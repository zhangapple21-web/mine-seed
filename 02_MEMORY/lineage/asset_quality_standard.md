# Asset Quality Standard — 文明资产质量标准

> **版本**: v1.0
> **日期**: 2026-07-13
> **来源**: skill-creator 的三个核心特质（可测试/可迭代/可调度）
> **目的**: 定义文明资产必须满足的质量要求，确保每个资产都能被路由器调度

---

## 核心定义

每个通过 Admission 的文明资产，都应该像 skill-creator 输出的技能一样，具备三个特质：

```
可测试（Testable）→ 有明确的验收标准
可迭代（Iterable）→ 有清晰的改进路径
可调度（Routable）→ 能被路由器识别和调用
```

---

## 质量标准详解

### 1. 可测试（Testable）

**定义**: 资产有明确的验收标准和验证方法。

**标准**:

| 检查项 | 合格 | 不合格 |
|--------|------|--------|
| 验收标准 | 有明确的 Acceptance Criteria | 没有或模糊 |
| 验证方法 | 有可执行的验证步骤 | 无法验证 |
| 证据链 | 有 supporting_evidence | 无证据或证据薄弱 |
| 重建提示 | 有 rebuild_hint | 无 |

**测试方法**:
- 公理/原则层：逻辑自洽性检查
- 治理/能力层：功能测试（是否能实现其声称的功能）
- 协议层：流程测试（是否能走完整个流程）
- 架构层：一致性检查（是否与现有架构兼容）

### 2. 可迭代（Iterable）

**定义**: 资产有清晰的改进路径和版本管理机制。

**标准**:

| 检查项 | 合格 | 不合格 |
|--------|------|--------|
| 缺口记录 | 有 gaps 字段记录已知问题 | 无 |
| 演化路径 | 有 parent_asset / superseded_by 记录 | 无 |
| 版本管理 | 有 distillation_date / distilled_by | 无 |
| 优先级 | 有明确的改进优先级（P0/P1/P2） | 无 |

**迭代方法**:
- 每个 gaps 应该有对应的改进计划
- 每次改进应该更新 lineage_index
- superseded_by 应该指向新资产

### 3. 可调度（Routable）

**定义**: 资产能被路由器识别、索引和调用。

**标准**:

| 检查项 | 合格 | 不合格 |
|--------|------|--------|
| 唯一标识 | 有 asset_id（NNN_asset_name） | 无或重复 |
| 分类清晰 | 有明确的 category | 无或错误 |
| 描述完整 | 有 name + description | 无或模糊 |
| 接口定义 | 有明确的输入输出（如果是能力/协议） | 无 |

**调度方法**:
- 资产应该能被 CSP Registry 索引
- 能力层资产应该有明确的 Capability 映射
- 协议层资产应该有明确的执行入口

## 质量评分

### 评分规则

每个特质满分 10 分，总分 30 分。

| 分数 | 等级 | 含义 |
|------|------|------|
| 25-30 | **★★★★★** | 高质量资产，可直接调度 |
| 20-24 | **★★★★☆** | 良好资产，轻微改进后可调度 |
| 15-19 | **★★★☆☆** | 中等资产，需要改进 |
| 10-14 | **★★☆☆☆** | 低质量资产，需要大幅改进 |
| 0-9 | **★☆☆☆☆** | 不合格资产，禁止入库 |

### 评分维度

| 维度 | 权重 | 评分标准 |
|------|------|----------|
| 可测试 | 33% | 验收标准明确（3）+ 验证方法可执行（3）+ 证据链完整（2）+ 重建提示（2） |
| 可迭代 | 33% | 缺口记录完整（3）+ 演化路径清晰（3）+ 版本管理完善（2）+ 优先级明确（2） |
| 可调度 | 34% | 标识唯一（3）+ 分类清晰（3）+ 描述完整（2）+ 接口定义（2） |

---

## 准入约束

### 质量阈值

- **合格线**: ≥ 15 分（★★★☆☆）
- **可调度线**: ≥ 20 分（★★★★☆）

### 禁止入库

- < 10 分的资产禁止入库
- 有严重缺陷（如重复、证据造假、违反约束）的资产禁止入库

### 暂缓入库

- 10-14 分的资产暂缓入库，改进后重新审查

## 审查结论

### 统计

| 等级 | 数量 | 占比 |
|------|------|------|
| ★★★★★ | 2 | 7% |
| ★★★★☆ | 14 | 50% |
| ★★★☆☆ | 12 | 43% |
| ★★☆☆☆ | 0 | 0% |
| ★☆☆☆☆ | 0 | 0% |

### 平均评分

- **可测试**: 7.32/10
- **可迭代**: 5.79/10 ← 最低
- **可调度**: 7.79/10
- **总分**: 20.9/30

### 主要问题

1. **可迭代性最差**（5.79/10）：大部分资产没有明确的改进路径和优先级
2. **r1-continuity-backup 来源的资产评分偏低**：缺少实现代码，可测试性差
3. **DFP-001 和研究原则库**：历史信息缺失，影响可迭代性

### 改进建议

1. **为每个资产补充改进计划**：在 gaps 字段中添加具体的改进步骤和优先级
2. **实现可测试的资产**：为能力层/协议层资产编写测试用例
3. **完善调度接口**：为每个资产添加明确的 Capability 映射

## 与 Admission Engine 的集成

在 Admission 六问的基础上，增加第七问：

```
7. Quality — 质量审查：可测试（≥7）+ 可迭代（≥5）+ 可调度（≥7）？
```

审查流程：
1. 准入六问 → 初步筛选
2. 质量审查（可测试/可迭代/可调度）→ 质量验证
3. 通过后进入 Repository
