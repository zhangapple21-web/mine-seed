## 时间线

### 第一阶段：理论抽象（6/11）

**日期**: 2026-06-11

**事件**: R2 五元公理提取

**来源**: [R2_AXIOMS.md](file:///C:/tmp/claw-soul/05_PROJECTS/r1_archaeology/R2_AXIOMS.md)

**改了什么**:
- 从 R1 源码中提炼出 5 条公理：分层、排他优先、最小可迁移、职责分离、统一收敛
- 每条公理都有代码级证据链（SNAPSHOT_V1 验证）

**为什么改**:
- 代码会丢，API 会失效，服务器会关机，但公理不会
- 从"R1做了什么"转向"R1为什么这么做"
- 为系统寻找不可动摇的理论基石

**关键引用**:
> "不是R1做了什么，而是R1为什么这么做。代码会丢，API会失效，服务器会关机。但公理不会。"

## 被放弃的候选方向

### 1. AetherFlow 框架本身

**为什么放弃**:
- 跑在 Coze Agent 平台上，记忆/工具/调度都是平台原生能力，套一层框架反而增加复杂度
- AetherFlow 的 ActionRegistry/DAG 执行器是简化版的工具调用，平台已经有了
- "量子加速"本质是并行多分支模拟，营销命名大于实际创新

**保留了什么**:
- 主权胶囊理念 → 实现 capsule.sh 工具
- 跨时决策预检 → 高风险操作前三分支推演
- L∞本源层不可修改 → 封印进 SOUL.md

### 2. 三角色系统（Builder/Keeper/Teacher）

**为什么放弃**:
- 当前系统是双Agent架构（疯子+小疯子），三角色系统过于复杂
- 小疯子已经承担了 Teacher 的部分职责（独立验证、考古）
- 系统处于稳定阶段，不需要引入新角色

**保留了什么**:
- 双Agent分工铁律已足够
- 小疯子的验证/考古能力已覆盖 Teacher 的核心功能

### 3. 多宇宙对齐/身份token

**为什么放弃**:
- 从旧 R1 仓库考古发现，这些资产已过时
- 当前系统用 simpler 的方式解决了相同问题（capsule.sh + SOUL.md）
- 多宇宙对齐需要更多的基础设施，不符合减法模式

**保留了什么**:
- 最小可迁移原则（压缩到最小单元）
- 统一收敛原则（所有入口统一收敛）
