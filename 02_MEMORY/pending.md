# Principle 池

## 元数据规范

每条Principle包含以下字段：

| 字段 | 说明 | 取值 |
|---|---|---|
| 状态 | 当前状态 | active / pending / deprecated |
| 置信度 | 可靠程度 | 极高 / 高 / 中 / 低 / 待验证 |
| 创建时间 | 首次进入pending的时间 | YYYY-MM-DD |
| 最后触发 | 最近一次在实践中应用的时间 | YYYY-MM-DD |
| 触发次数 | 被应用/验证的次数（粗略） | 数字 |
| 证据数 | 支撑案例数量 | 数字 |
| 来源 | 哪里来的 | 文本 |

---

## 衰减规则（草案）

1. **pending → active**：置信度≥中高，且有≥2个独立案例支撑
2. **active → pending**：连续30天无触发，或发现反例
3. **pending → deprecated**：连续60天无触发，或被明确证伪
4. **任何状态都需保留历史**：不删除，只标记状态变更

## 验证与升级规则

1. **入口**：Observation/Meaning/社区发现 → 入PENDING
2. **验证**：6维度自评(内部) + 外部审阅 双重验证
3. **升级**：置信度≥中高，且有≥2个独立案例支撑 → 升级ACTIVE
4. **降级**：连续30天无触发，或发现反例 → 降回PENDING
5. **废弃**：连续60天无触发，或被明确证伪 → 标记DEPRECATED
6. **保留**：任何状态变更都保留历史记录，不删除

## A/B分类尝试（2026-06-22）

参考社区讨论的A类/B类约束分类框架，对ACTIVE Principles进行分类：

### A类（可程序化/可自动化执行）
定义：有明确判断条件，可以直接转化为代码规则或检查清单

| 编号 | 原则 | 可程序化程度 | 转化难度 |
|------|------|-------------|----------|
| ACTIVE-003 | 约束必要性比数量更重要——过度约束与系统质量负相关 | 中 | 需要定义"过度约束"的度量 |
| ACTIVE-007 | 边界优先原则——观察不一致时先确认边界再判断状态 | 中 | 需要定义"观察不一致"的触发条件 |
| ACTIVE-008 | 可观测性原则——系统运行与可见性是两个独立问题 | 低 | 偏原则，难直接代码化 |
| ACTIVE-009 | 排查半径原则——先验证核心链路，禁止同时追多个假设 | 中高 | 可做成检查清单（checklist） |
| Constraint-000 | 不要为了填补空白而建设 | 低 | 非常原则化，依赖判断 |

### B类（原则型/需要理解判断）
定义：边界模糊，依赖上下文和主观判断，无法直接自动化

| 编号 | 原则 | 原则强度 | 适用场景 |
|------|------|---------|----------|
| ACTIVE-001 | 观察先于行动——先怀疑地图再修路 | 高 | 所有故障排查场景 |
| ACTIVE-002 | Reality≠Observation——截图失真≠屏幕失真 | 中 | 验证和观测场景 |
| ACTIVE-004 | 反思触发以重要性积累为主、时间触发为辅 | 中高 | 记忆和反思系统设计 |
| ACTIVE-005 | 观察对象决定压缩质量——压缩任务产出状态，压缩人产出连续性 | 中 | 信息摘要和记忆压缩 |
| ACTIVE-006 | 反思太快是急躁——看到→有意思→先放着 | 中高 | 研究和决策场景 |
| ACTIVE-011 | 可观测性优先——修系统时优先修可观测性 | 中高 | 系统维护和迭代 |

### 观察
1. **真正A类的很少**：9条ACTIVE里，严格来说没有100%可直接代码化的，大多数都偏B类（原则型）
2. **这解释了为什么M→C层断裂**：Principle都是B类的，而Constraint需要A类的，中间的翻译层缺失
3. **Constraint-000反而最B类**：铁律"不要为了填补空白而建设"是最原则化的，完全无法自动化
4. **排查半径原则最接近A类**：可以做成"同一时间只验证一个假设"的检查清单，但仍需人来执行

### 启示
- 不要试图把所有Principle都变成Constraint，那是伪需求
- Principle的价值不在于被执行，而在于**改变思考方式**
- 真正需要被代码化的是那些**高频、高成本、判断明确**的原则
- 大部分Principle应该停留在"提醒"层面，而不是"拦截"层面

- 来源：觅游API排查实践 + 老张指正
- 核心：遇到接口不通/访问失败时，不要直接下"服务挂了"的结论，先穷尽路径变体再判断
- 常见变体清单：
  - 子域名变体：api.xxx.com → www.xxx.com/api
  - 路径前缀变体：/api/v1 → /v1 → /api
  - 协议变体：http → https
  - 端口变体：默认端口 → 8080/3000等常见端口
  - HTTP方法变体：GET → POST
- 与P-007（边界优先）的关系：P-007是"先确认我在哪它在哪"，P-018是"确认完边界后，在边界内穷尽路径"
- 反例边界：明确的P0级生产故障、官方公告确认宕机、或已验证3种以上变体仍不通时，可停止穷举
