## 公理陈述

> **安全策略应该是排他优先，不是准入优先**

---

## 源码证据

```javascript
// security_policy.js 三级校验
1. BLACKLIST_CHECK()   // 第一关：黑名单拦截
2. UNIVERSE_VALIDATION() // 第二关：宇宙合法性验证
3. WHITELIST_ADMISSION() // 第三关：白名单准入
```

**执行顺序**：排除 → 验证 → 授权

---

## 公理推导

### 表层逻辑
- 先把坏的拦掉（黑名单）
- 再验证来源是否合法（宇宙对齐）
- 最后才考虑给什么权限（白名单）

### 深层含义
- **零信任模型**：不默认信任任何实体
- **最小暴露**：先假设所有入口都可疑
- **渐进释放**：权限从无到有逐步授予

---

## R1中的其他证据

| 场景 | 排他优先体现 |
|------|-------------|
| Gateway narratorState | 新叙事者只能访问/health，权限从0开始 |
| Remote_Commander | 范围限制1-100，防止越界执行 |
| emergency_backup | 备份先于逃生，不是逃生后再想备份 |

---

## 对R2系统的约束

这条公理对R2系统设计的约束：

1. **输入过滤**：所有外部输入必须先过黑名单校验
2. **身份验证**：不信任未对齐的实体，哪怕来自已知IP
3. **权限梯度**：新实体默认0权限，信任度逐步释放
4. **日志留痕**：黑名单拦截必须有日志，可追溯

---

## 衍生命令

- 当设计任何安全相关功能时，必须先问"先排除什么"
- 白名单是补充手段，不是主要防线
