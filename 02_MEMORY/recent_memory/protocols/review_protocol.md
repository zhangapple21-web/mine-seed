# 产出验证协议 v1.0

## 原则
节点会换，协议留下。不依赖具体模型/API/实现。

## 触发条件
1. 知识班产出 → 自动触发
2. Principle候选 → 提出后24h内触发
3. 重要决策/产出 → 主观判断触发（任何维度自评<0.7时）

## 验证步骤

### Step 1: 自评（内部）
6维度各0-1：准确性/完整性/效率/稳定性/可执行性/合规
- 任何维度<0.5 → 必须标注"不确定"
- 综合<0.7 → 标注低置信度，说明哪维不足

### Step 2: 外部审阅（军师）
将产出+6维度自评提交给外部审阅者。
审阅者可以是：任何可用模型/子session/社区评论/老张
审阅者必须独立评分，不看内部自评结果。
外部审阅只负责：按6维度打分+指出最弱维度+给改进建议

### Step 3: 仲裁
| 情况 | 处理 |
|------|------|
| 内部≥0.7 且 外部≥0.7 | 通过 |
| 内部≥0.7 但 外部<0.7 | 以外部为准，标注差距 |
| 内部<0.7 且 外部<0.7 | 不通过，需改进后重审 |
| 双方最弱维度一致 | 高置信问题 |
| 双方最弱维度不同 | 扩展检查范围 |

### Step 4: 归档
- 审阅记录存 research/reviews/review_YYYYMMDD_HHMM.json
- 含：产出标题/内部评分/外部评分/仲裁结论/改进动作
- Principle升级/推翻记录在 principles/pending.md

## Principle生命周期

```
观察/发现 → PENDING(提出) → 24h内验证 →
  ├─ 置信度≥0.6 → ACTIVE(带修正/反例边界)
  ├─ 置信度<0.6 → PENDING(标注验证路径，等待)
  └─ 强反例且无法修正 → REJECTED(保留记录)
→ ACTIVE每季度回顾 → 维持/修正/降级
```

## 元认知（2026-06-20 老张校正）

这次升级的不是4条Principle，而是验证协议本身。

以前经验仓是在收集观点，现在开始收集经过反例审查的结构。

ACTIVE不是真理，而是当前证据下最可靠的工作假设。

PENDING也不是失败，而是在提醒：系统知道自己还不知道什么。
