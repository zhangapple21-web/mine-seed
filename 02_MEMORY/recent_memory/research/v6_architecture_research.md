# V6架构研究笔记：从执行系统到治理系统的跃迁

**日期**: 2026-06-22
**研究对象**: 矿场V6 RFC系列文档
**研究维度**: 架构演化、约束体系、治理机制
**状态**: 进行中

---

## 核心洞察：V6不是版本升级，是物种演化

```
V1 = 能跑
V2 = 能扩展
V3 = 能协作
V4 = 能积累经验
V5 = 能持续运行
V6 = 能治理自己
```

V1-V5解决的是**怎么干活**，V6开始解决的是**谁能干活、什么时候干活、为什么干活**。

触发点：80个Session堆积事件。这不是V5能力不足，是**执行权失控**——谁都能spawn，谁都能占资源，结果没人管=瘫痪。

> 这是Mining System进入Ecosystem的分水岭。

---

## 三大核心协议

### 1. Scheduler：执行权唯一坍缩点

**核心原则**：Execution Must Collapse Through One Gateway.

所有执行意图必须经过唯一的调度器坍缩成有序的执行序列。

```
之前：分散执行
Host → 矿工A（直接spawn）
Host → 矿工B（直接spawn）
... → 资源竞争 → 死锁

之后：统一调度
Host → Scheduler(CCO) → Worker Pool
```

**任务优先级体系（A > C > B）**：
- A类（生产）：必须执行，可抢占
- C类（维护）：低频重要，A类来了让路
- B类（探索）：有更好，没也行，永远让路

**对研究域的启发**：
我们在Constraint调用链里发现的"语义层断裂"问题，本质上也是控制权分散——O→E→C每一层都在解释约束，但没有统一的坍缩点。研究域的Principle体系也需要一个"意义坍缩点"，否则每层都在重新解释，最后执行层收到的和最上层想的完全不是一回事。

### 2. Constraint Engine：经验的可执行化

**核心闭环**：Observation → Experience → Constraint → Scheduler

约束是经验的可执行形式。经验不该停在文档里，它必须回流成约束，自动注入调度器。

| 阶段 | 回答问题 | 示例 |
|------|----------|------|
| Observation | 发生了什么 | gh_4o在canonical_v2上5场1胜 |
| Experience | 所以呢 | gh_4o在这类任务上不可靠 |
| Constraint | 那怎么办 | FORBID gh_4o 接收canonical_v2任务 |

**约束生命周期**：
```
draft → active → expired → archived
```

这和我们研究域的Principle三态（pending/active/deprecated）本质上是同构的。

**对研究域的启发**：
我们现在有O→E→P（Principle）的路径，但缺少P→C的工程化闭环。Principle还停留在"文档"阶段，没有变成可执行的约束。这是下一步需要打通的。

### 3. Worker Registry：能力画像与动态调度

**核心思想**：矿工必须注册才能上岗，注册即声明能力，考核即决定去留。

每个矿工有完整的能力档案：
- 各任务的成功率（SR）和延迟
- 活跃约束
- 运行统计
- 等级（primary/reserve/candidate）

调度器根据任务类型和矿工能力画像做最优匹配，而不是盲派。

**对研究域的启发**：
研究域也可以建立"Worker能力画像"——不是矿工，是"研究视角"或"分析框架"。不同的研究问题，适配不同的分析框架。比如：
- 失效分析 → 假设树框架
- 原则提炼 → 正反案例验证框架
- 边界探索 → 压力测试框架

## 四条可迁移的Principle

从V6架构中，可以提炼出四条适用于研究域的原则：

### P-010：控制权坍缩原则
任何分布式系统，当复杂度跨过临界点后，必须建立唯一的控制权坍缩点。否则必然出现资源竞争、死锁和全局失序。

**验证案例**：
- 80个Session堆积 → 引入Scheduler后解决
- Constraint语义层断裂 → 本质是缺少意义坍缩点

**适用边界**：
- 复杂度未跨过临界点时，分散执行效率更高
- 坍缩点本身可能成为单点故障，需要超时释放机制兜底

### P-011：经验闭环原则
经验如果不能回流成可执行的约束，就等于没有闭环。停在文档里的经验只是资讯，不是系统能力。

**验证案例**：
- gh_4o连续8败，经验写了但还在派活 → 没有闭环
- 80个Session堆积，写了复盘但还能随便spawn → 没有闭环

**适用边界**：
- 严重故障（数据丢失、安全风险）→ 自动激活约束
- 性能问题 → 2次观察后激活
- 轻微异常 → 观察记录，不急于约束

### P-012：能力注册原则
执行单元必须注册声明能力，调度器基于能力画像派单，而不是盲派。没有花名册的调度是碰运气。

**验证案例**：
- 矿工盲派 → 成功率不稳定
- Worker Registry建立后 → 基于SR做最优匹配

### P-013：分级治理原则
任务必须分级，资源必须向高优先级倾斜。B类任务永远给A类让路，这不是歧视，是系统生存的必要条件。

## 对研究域的三个行动建议

1. **建立意义坍缩点**：解决Constraint语义层断裂问题的根本方案，是在Principle层建立唯一的"意义解释权"，而不是每层各自解释。

2. **打通P→C工程闭环**：Principle不能只停留在文档里，需要有机制让Principle自动转化为可执行的Constraint。这可能是研究域下一个核心课题。

3. **建立研究方法注册表**：把不同的研究框架（假设树、案例验证、边界探测等）注册成"研究Worker"，根据研究问题类型自动适配最优框架。
