## 现象

Cloudflare Workers Build 之外，真正的公网桥问题出现在 zrok 层：

- 旧保留名 `data-r1` 无法正常恢复
- share 相关进程重启后持续失败
- systemd 旧服务反复重启，刷出大量垃圾日志

对外表现为：

- 旧桥失效
- 新桥需要重新建立
- 本地文档与验活脚本仍指向旧链路

---

## 三层根因

### 1. 主因：zrok reserved name 机制缺陷

`data-r1` 是保留名。  
当 share 进程崩溃后，zrok 服务端仍认为该 share 存活。

随后 agent 重启并试图重连，zrok API 返回：

```text
409 "name already in use"
```

结果：

```text
重连失败
↓
无限 retry
↓
永远无法恢复
```

这不是单纯的“网络抖动”，而是 reserved name 的状态残留问题。

### 2. 雪上加霜：废弃 systemd 服务持续刷噪声

存在一个旧的：

```text
zrok-share.service
```

它仍在尝试启动旧的 `r1-oneapi` / 旧 share 逻辑，并因为同样的名字冲突每 15 秒重启一次，累计刷出 2100+ 次失败日志。

结果：

- 把真正问题淹没在噪声里
- 增加排查成本
- 形成“旧配置幽灵”

### 3. 环境事实漂移：旧名字已被清理，但本地/文档仍沿用旧口径

此前为了修 dashboard，旧名字 `data-r1` 已经被清理。  
但：

- 本地自愈脚本 `tunnel_guardian.sh` 仍可能使用旧名
- 仓库文档和验活脚本仍保留旧入口或旧服务名

这导致：

```text
现实已经迁移
↓
记忆没有同步
↓
后续修复继续往旧世界发力
```

---

## 方法论沉淀

### 1. 名字冲突不是“偶发错误”，而是治理问题

当系统依赖“保留名”时，名字本身已经成为基础设施。  
它的生命周期必须进入治理范围，而不能只当作一个配置值。

### 2. 废弃服务必须真正退场

只要旧服务还能自动重启，它就还是系统的一部分。  
“忘了删”本身就是基础设施风险。

### 3. 修复必须同步三层

修复公网桥时，至少要同步：

1. 现实层：新服务、新入口、新端口
2. 记忆层：文档、README、案例记录
3. 自愈层：检查脚本、守护脚本、告警规则

少同步一层，问题就会回来。
