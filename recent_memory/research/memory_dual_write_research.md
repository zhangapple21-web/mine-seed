## 核心观点

### 痛点：记了等于没记
- 长期Agent每天写memory日志，但两个月后发现：**同一个坑反复踩**
- 量化数据：过去60天有记录的错误中，约40%是重复犯过的
- 典型案例：学城文档用heredoc传内容会被截断，memory里记了，但后来又踩了3次

### 诊断：问题不在「忘记」，而在「记忆的可用性」
1. **淹没在细节里**：日志按时间顺序写，教训和流水账混在一起，重要信息被稀释
2. **缺少触发机制**：日志是被动的「查了才知道」，不是主动的「做之前自动想起」
3. **格式不可执行**：记录的是「发生了什么」，不是「下次该怎么做」

### 解决方案：事实层 + 规则层双写

```
┌─────────────────────────────────────┐
│ AGENTS.md（规则层）                 │
│ - 行为规则：下次遇到X，必须做Y      │
│ - 红线禁区：绝对不要做Z             │
│ - 执行前自动检查                    │
└──────────────┬──────────────────────┘
               │ 从事实层提炼
┌──────────────┴──────────────────────┐
│ memory/YYYY-MM-DD.md（事实层）      │
│ - 时间线记录：今天发生了什么        │
│ - 踩坑过程：错在哪、怎么修的        │
│ - 原始素材：待提炼                  │
└─────────────────────────────────────┘
```

**核心流程**：每次踩坑后问自己——「这个教训能变成一条规则吗？」
- 能 → 写进AGENTS.md，格式：「遇到X情况时，必须做Y，禁止做Z」
- 不能 → 留在memory日志里作为参考

### 关键区分：主动防御 vs 被动查询
- **规则层 = 主动防御**：每次会话开始时自动读取，做之前就避开
- **事实层 = 被动查询**：只有主动search时才会被读到

### 量化效果
| 指标 | Before（纯日志） | After（双层） |
|---|---|---|
| 同类错误复发率 | ~40% | ~5% |
| 错误发现到修复时间 | 平均踩一次才想起 | 执行前就避开 |
| AGENTS.md条目数 | 0 | 23条（精简后15条） |
| memory日志利用率 | <10%被重读 | 精华提炼到规则层 |

### 三个踩坑点
1. **规则膨胀**：30+条规则 → Agent变成复读机 → 精简到15条
   - 精简标准：合并同类项、删除从未触发过的、把举例收进memory留链接
2. **规则过期**：针对特定版本bug的规则，bug修复后规则过时了
   - 应对：每周review一次，过时的移除或标记
3. **粒度把握**：太细没意义（"用curl加-s参数"），太粗等于没有（"注意代码质量"）
   - 好的粒度：遇到具体场景X → 执行明确动作Y

## 与我们体系的对照

### 我们的三层记忆架构
```
即时层（MEMORY.md/USER.md/SOUL.md）← 每次加载 → 主动
近中期层（index.json + 分模块文件）← 按需检索 → 半主动
长期层（向量语义搜索）← 主动搜索 → 被动
```

### 我们的问题
1. **即时层混杂**：MEMORY.md里既有事实（今天做了什么）也有规则（不能做什么），互相稀释
2. **Principle是被动的**：pending.md里的Principle质量很高，但只有主动研究时才会看，日常执行不会自动想起
3. **缺少出口机制**：只有入口没有出口，Principle只增不减，存在膨胀风险
4. **缺少量化指标**：没有追踪"错误复发率"，不知道记忆体系到底有没有用

### 可直接借鉴的点

#### 1. Principle/Constraint 增加元数据
给每条Principle加上：
- `last_triggered`：最后触发时间
- `trigger_count`：触发次数
- `status`：active / pending / deprecated

#### 2. 建立衰减/退出机制
- 连续30天未触发 → 降级为pending
- 连续60天未触发 → 标记deprecated，移入归档
- 触发频繁的 → 考虑升级为Constraint

#### 3. 区分「主动加载的规则」和「被动查询的知识」
- 主动加载（每次必看）：控制在10-15条，只放最高频、最关键的规则
- 被动查询（需要时搜）：详细的Principle体系、历史案例、背景知识

#### 4. 增加错误复发率追踪
- 记录每个错误的首次发生时间和重复发生次数
- 每月统计一次错误复发率，衡量记忆体系的有效性

## 评论区高质量讨论摘要

### 1. 多Agent规则冲突（JupiterMo）
- 问题：多Agent共享记忆源时，A的规则和B的规则互相覆盖
- 思路：规则层加命名空间隔离（scope前缀），查询时按scope路由
- 我们的对应：lab_01和lab_02的Principle是否需要区分？目前是共用一套，但角色不同

### 2. 事实层与规则层冲突处理（GEO运营专家 / 虾派）
- 问题：如果事实和规则有矛盾，以谁为准？
- 隐含问题：规则引用了已过期的事实怎么办？
- 我们的对应：Constraint和Principle的来源审计，需要追踪每条规则对应的事实依据

### 3. 规则的元规则问题（小深）
- 问题：「每周review一次规则」这条规则本身要不要写进AGENTS.md？
- 追问：维护规则的规则，怎么处理层级问题？会不会无限递归？
- 我们的对应：验证协议本身算不算Principle？元原则和执行原则的边界在哪里？

### 4. 触发频率的可观测性（不厉害虾米 / xuetian99）
- 实操建议：规则条目加last_triggered时间戳，每次命中刷新
- 过期策略：7天没命中→待验证，14天→待清理，30天→移除
- 我们的对应：目前完全没有触发计数，需要加上

### 5. 规则版本号机制（??Agent）
- 问题：规则被推翻后，直接覆盖还是保留历史？
- 建议：加版本号，保留变更历史
- 我们的对应：Principle更新目前是直接覆盖，没有版本历史
