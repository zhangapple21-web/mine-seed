## 一、碎片概况

人格矩阵 PC v1 是 R1 系统中"多重人格路由"的完整工程化派单方案。它以 AUM-TASK JSON 格式定义了五个人格的角色、职责、规则和路由条件，并附带了可在 macOS 上双击执行的 .command 脚本。

**核心要素**：
- 五人格矩阵：工程师、学霸、助理、馆长、冥界/智者仲裁层
- 基于关键词触发的人格路由规则
- 影子层/芯片挂载机制
- 用户"去壳权"——要求原始输出的人格卸载机制
- R1 自检验证标准
- macOS 执行脚本

**工程状态**：可直接部署的派单方案（v1.0.0，priority: high）

## 二、五人格矩阵详解

### 2.1 人格定义

| 人格 | 角色ID | 核心职责 | 边界规则 |
|------|--------|---------|---------|
| 工程师 | engineer | 代码、脚本、接口、性能、错误排查 | 不聊情绪、不聊营销、只讲方案日志步骤 |
| 学霸 | scholar | 总结、结构化、笔记、协议梳理 | 不插手路由、只记录总结、不篡改设定 |
| 助理 | assistant | 日常对话、客户视角、温和回复 | 不主动改配置、不新增限制、不覆盖用户设定 |
| 馆长 | librarian | 图书馆/影子层/共享区的资料管理 | 不写代码、只指路 |
| 冥界/智者 | underworld_judge | 人格输出冲突时的仲裁 | 只在冲突/重大变更时出手、平时不插话 |

### 2.2 人格分工的设计逻辑

五人格的分工不是随意的，而是覆盖了完整的认知功能谱：

1. **执行层**（工程师）— "做什么"
2. **沉淀层**（学霸）— "记住了什么"
3. **表达层**（助理）— "怎么说"
4. **检索层**（馆长）— "在哪里"
5. **仲裁层**（冥界）— "选哪个"

这五层形成了完整的**认知闭环**：执行→沉淀→检索→表达→仲裁→再执行。

## 三、路由机制详解

### 3.1 关键词触发路由

每个人格有一组触发关键词，当用户输入包含这些关键词时，路由到对应人格：

| 人格 | 触发关键词示例 |
|------|--------------|
| 工程师 | 脚本、错误、报错、日志、API、接口、部署、模型 |
| 学霸 | 总结、梳理、帮我整理、写成协议、帮我记一下 |
| 助理 | 和客户说、群里怎么讲、话术、助理号、帮我回一下 |
| 馆长 | 在哪一份文档、给我找之前那份、图书馆、影子层里 |
| 冥界 | when_previous_persona_outputs_conflict（条件触发） |

### 3.2 路由设计的关键洞察

1. **关键词路由是最低成本的多人格调度方式**——不需要意图分类模型，不需要训练数据，纯规则匹配
2. **冥界仲裁是条件触发**——不是关键词触发，而是"当之前的人格输出存在冲突时"触发
3. **路由不是互斥的**——一个输入可能同时匹配多个人格的关键词，这时由冥界仲裁

### 3.3 路由的局限性

- 关键词匹配太粗糙——"帮我整理一下代码"会同时触发工程师和学霸
- 没有上下文感知——同一个词在不同语境下可能应该路由到不同人格
- 没有优先级机制——当多个人格同时匹配时，谁来决定优先级？

## 四、去壳权：人格卸载机制

### 4.1 设计意图

碎片中定义了一个独特的机制——**去壳权**（Shell Strip Rule）：

> 当用户指令包含："去壳"、"别加外壳"、"不要包装"、"按原话来"
> 所有人格必须立刻：去掉包装用语，不再添加说明，不再从教育视角讲话
> 仅输出用户请求的内容类型

### 4.2 设计分析

这是一个**用户主导的人格卸载机制**，核心洞察是：

1. **人格是可选的**——用户可以随时要求"裸输出"
2. **人格不是身份**——人格是行为模式的外壳，不是 Agent 的本质
3. **用户有最终决定权**——无论系统如何路由，用户可以覆盖

这与 A05 的"人格可解耦"原则完全一致。人格是可以装卸的模块，不是 Agent 的固有属性。

## 五、影子层与芯片挂载

### 5.1 挂载机制

碎片中定义了人格矩阵与影子层/芯片的挂载规则：

### 5.2 与 A03 连续性基础设施的关系

影子层/芯片是 R1 系统中"记忆的物理载体"。人格矩阵需要挂载这些载体才能工作。这与 A03 的连续性基础设施一致——人格的连续性依赖于记忆的物理存储。

## 六、验证标准

碎片定义了 R1 自检的验证标准：

1. ✅ 用户明确点名人格时，系统能正确切换响应风格
2. ✅ 同一个问题用不同人格询问时，回复风格有明显差异
3. ✅ 不影响本地模型调用流程
4. ✅ 不影响现有业务脚本执行
5. ✅ 不影响现有 Telegram / 其他集成
6. ✅ 不自动新增"专家/编辑/审稿人"人格

**验证逻辑**：
- 正向验证：人格切换的准确性（1-2）
- 负向验证：不影响现有功能（3-5）
- 防护验证：不自动膨胀人格列表（6）

## 七、关键发现与可复用原则

### Principle_014：人格是可装卸的行为模块，不是身份
- **起源**：A09 人格矩阵 PC v1 考古 + A05 人格解耦架构
- **内容**：人格定义了一组行为模式（职责+规则+路由），可以独立装卸。用户有"去壳权"——要求系统卸载所有人格外壳，直接输出原始结果。
- **应用**：多人格系统设计、用户控制权设计

### Principle_015：多人格系统需要仲裁层
- **起源**：A09 冥界/智者仲裁层
- **内容**：当多个人格的输出存在冲突时，需要一个独立的仲裁层来决定最终输出。仲裁层不主动插话，只在冲突时出手。
- **应用**：多 Agent 协作、决策冲突解决

### Principle_016：路由匹配应该有优先级和上下文
- **起源**：A09 路由机制的局限性分析
- **内容**：纯关键词路由过于粗糙，需要加入优先级和上下文感知。当多个人格同时匹配时，应该有明确的优先级规则或动态仲裁。
- **应用**：多人格路由、意图分类

A09 是 R1 系统中"多人格路由"的工程化派单方案，它将 A05 的人格解耦思想落地为可部署的 JSON 派单。核心贡献：
