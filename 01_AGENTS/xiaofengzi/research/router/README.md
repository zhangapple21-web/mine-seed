# 路由层说明

**状态**: v0.1 人工路由阶段

---

## 核心学习闭环（最高优先级）

这是研究域最值钱的流程，所有其他模块都为这个闭环服务。

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  发现失效   │────▶│  M02 失效   │────▶│  M04 Constraint │
│  / 异常     │     │   分析      │     │     提案      │
└─────────────┘     └─────────────┘     └──────┬──────┘
       ▲                                        │
       │                                        │
       └────────────────────────────────────────┘
                   行为改变，减少未来失效
```

**触发条件**：任何模块发现失效/异常/错误时，优先走这个闭环，而不是继续发现新东西。

**优先级**：P0 > 其他所有研究任务。

---

## 当前路由方式

目前由知识早班/午班人工判断研究类型，分发给对应模块。
后续目标：实现半自动化路由，根据研究问题的关键词自动匹配模块。

---

## 模块优先级与路由规则

| 优先级 | 模块 | 接到的问题/发现 | 典型触发源 |
|--------|------|----------------|-----------|
| **P0** | M02 失效分析 | 某个策略/Worker/信号表现异常、突然不灵了、踩了坑 | 生产异常上报、M01验证发现失效、人工踩坑 |
| **P1** | M04 Constraint提案 | 从失效/成功中提炼出了可复用的规则/原则 | M02输出结论、M01验证通过、R1考古发现 |
| **P2** | M03 Worker画像 | 想知道某个Worker擅长什么、边界在哪里、什么场景该用什么 | 新Worker上线、Worker表现波动 |
| **P3** | M06 R1考古 | R1的某个设计思路/架构原则好像能复用 | 日常考古、碎片挖掘、架构问题 |
| **P4** | M05 同构检测 | 两个研究结论好像说的是一回事、可能重复建设 | 多个模块产出后、定期清理 |
| 基础 | M01 信号验证 | 某个信号好像有效、想验证一下 | 日常观察、外部输入、回测发现 |

> **路由铁律**：当P0和P3同时有任务时，先做P0。先把坑填上，再去挖新东西。

---

## 跨模块协作流程

### 流程1：失效学习闭环（P0→P1）
1. 任何地方发现失效 → 进入M02做深度分析
2. M02输出根因结论 → 自动触发M04 Constraint提案
3. M04输出草案 → 经过观察期验证 → 进入全局约束体系
4. 约束生效 → 指导未来行为 → 减少同类失效

### 流程2：信号验证闭环
1. 发现疑似有效信号 → M01验证
2. 验证通过 → 可触发M04提炼为约束
3. 验证不通过 → 可触发M02分析为什么无效

### 流程3：Worker能力闭环
1. Worker表现波动 → M03画像更新
2. 发现Worker在某类场景下必跪 → 触发M02失效分析
3. 提炼为"什么场景不能用什么Worker"的约束 → 进入M04

## 一致性审查

所有模块产出后，必须经过一致性审查：
1. 是否与已有Principle冲突？
2. 是否与已有Constraint冲突？
3. 是否与之前的研究结论矛盾？

如果有冲突，标记为"待调和"，不直接入藏。
