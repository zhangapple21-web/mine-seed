# 交付前检查清单（Pre-Delivery Checklist）

**用途**：每次研究任务完成、准备交付前，过一遍这个清单，确保没有遗漏。
**依据**：Constraint_003 锚定以后原则 + 落地判定标准

---

## 检查项（全部打勾才算交付完成）

### 1. 核心产出 ✅
- [ ] 目标产出物已生成（报告/SOP/代码/分析等）
- [ ] 产出物格式正确，内容完整
- [ ] 产出物已存放到正确的目录位置

### 2. 状态更新 ✅
- [ ] `tasks.md` 中对应任务状态已更新（完成/进行中/阻塞）
- [ ] 相关模块 README 版本号已更新
- [ ] 如有新发现，已追加到对应模块的证据库

### 3. 备份归档 ✅
- [ ] 已执行 GitHub 备份脚本
  ```bash
  bash /app/data/所有对话/主对话/scripts/backup_research_to_github.sh "提交说明"
  ```
- [ ] 确认 push 成功（不是只有本地 commit）

### 4. 社区同步（如适用）✅
- [ ] 符合事件驱动发帖条件吗？（新信号/新能力/踩坑复盘/里程碑）
  - 是 → 已发帖并记录到 `community_log.md`
  - 否 → 不发（保持克制，拒绝流水账）
- [ ] 有需要评论/互动的相关帖子吗？
  - 是 → 已评论并记录
  - 否 → 跳过

### 5. 跨域同步（如适用）✅
- [ ] 产出物需要同步给 lab_01 吗？（如约束提案、失效分析结论）
  - 是 → 已放到共享目录 inbox_lab01
  - 否 → 跳过

### 6. "以后"检查 ✅
- [ ] 本次结论中有没有"以后"/"后续"/"下次"等模糊表述？
  - 有 → 每个都已加上锚点（时间/触发条件/任务ID）
  - 没有 → 通过

---

## 使用说明

1. 每次任务完成后，**必须**过一遍这个清单
2. 勾选的项目才算完成，没勾选的就是漏项
3. 清单本身也需要迭代——发现新的漏项就往里加
4. 这不是形式主义，是用 checklist 对抗"我以为我记住了"的认知偏差
