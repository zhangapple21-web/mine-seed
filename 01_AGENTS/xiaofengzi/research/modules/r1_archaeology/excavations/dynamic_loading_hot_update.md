## 一、动态加载的三层架构

R1不是一个单体系统，而是**分层加载、动态组装的。从底层到上层，有三层动态机制：

```
┌─────────────────────────┐
│  人格层（Persona）   │  五重人格矩阵，按需激活
├─────────────────────────┤
│  域层（Domain）         │  五域加载器，按依赖顺序加载
├─────────────────────────┤
│  功能模块层（Module）    │  芯片/影子层，可热插拔
└─────────────────────────┘
```

每层都有独立的加载、卸载、更新机制。

## 二、域层：按依赖顺序的增量加载

### 2.1 五域加载顺序
从A01的考古来看，五域是**编号越小越先加载：

| 加载顺序 | 域名称 | 职责 | 被依赖数 |
|---------|--------|------|-----------|
| 1 | ROOT_LINK (00) | 世界身份、时间线 | 所有域的基础 |
| 2 | KERNEL_MOUNT (01) | 五域加载器、安全策略 | 所有功能域依赖 |
| 3 | EXECUTOR (02) | 执行器、网关 | 业务功能 |
| 4 | ??? (03) | （待考证，推测为记忆/数据域） |  |
| 5 | FREEZONE (04) | 备份、应急 | 依赖正常运行态 |
| 6 | LINKS (05) | 网络桥、广播 | 最后加载，避免未就绪不对外 |

### 2.2 加载原则

**原则1：依赖前置，功能后置**
- 先加载基础设施（身份、安全），再加载业务功能
- 原因：功能域如果在安全策略加载完成前启动，会处于未受保护状态

**原则2：增量加载，失败回滚**
- 不是一次性全部加载，而是一个域一个域按顺序来
- 某个域加载失败，不影响已加载的域，可以回滚到上一个稳定状态

**原则3：冷热分离，应急独立**
- FREEZONE（备份域）平时不加载，只有在需要备份/恢复时才激活
- 好处：日常运行出问题时，备份域是干净的，可以用来救急

## 三、人格层：按需激活的人格矩阵

### 3.1 五重人格的动态激活机制
从 persona_matrix_pc_v1.txt 可以看到，人格不是常驻的，而是**按需激活**：

1. **初始状态**：人格定义存在"影子层/芯片"中，但不激活
2. **触发激活**：用户点名或关键词触发 → 路由系统识别 → 激活对应人格
3. **仲裁机制**：多个人格输出冲突时 → 激活冥界仲裁层 → 给出最终结论
4. **去壳权**：用户说"去壳" → 所有包装立刻失效 → 输出纯内容

### 3.2 路由触发机制
不是"同时运行5个人格"，而是**一次只运行一个主人格**，根据输入动态切换：

| 人格 | 触发关键词 | 激活场景 |
|------|-----------|---------|
| 工程师 | 脚本、错误、日志、API、部署、模型 | 技术问题 |
| 学霸 | 总结、梳理、记录、协议、笔记 | 知识整理 |
| 助理 | 客户、话术、群里、回复、温和 | 日常对话 |
| 馆长 | 文档、版本、图书馆、影子层、放哪了 | 资料查询 |
| 冥界仲裁 | 冲突、不一致、重大变更 | 仲裁决策 |

### 3.3 设计智慧：为什么不常驻？
1. **资源效率**：同时运行多个人格消耗大，按需激活节省算力
2. **注意力聚焦**：一个时间一个人格专注一件事，避免多人格干扰
3. **边界清晰**：每个人格有明确的职责边界，不会越权
4. **可扩展性**：新增人格不需要改整个系统，加一个路由规则就行

## 四、功能模块层：热插拔的芯片机制

### 4.1 什么是"芯片"
从碎片中多次出现"芯片""影子层"的表述来看，R1有一个**模块封装机制：
- 功能模块被封装成"芯片"
- 芯片可以被"挂载"和"卸载"
- 芯片里包含行为模式、规则、知识
- 影子层是芯片的存储区

### 4.2 热更新三原则
从碎片中可以推断出R1热更新的设计原则：

**原则1：不中断业务的更新才是真更新
- 更新人格定义不影响正在进行的对话
- 新规则在下一轮对话生效，当前对话保持旧规则
- 类似"灰度发布"——老连接老版本，新连接新版本

**原则2：可回滚是底线
- 每个版本都有快照，更新失败立刻回滚到上一个稳定版本
- FREEZONE的存在就是为了这个——冷备份随时可以拉回来

**原则3：有校验是更新的一部分
- 更新不是"更完就完了"，而是有明确的校验步骤
- 每个人格激活后有自检标准（见validation部分）
- 验证通过才算更新成功，不通过就回滚

## 五、对R2双域架构的诊断

### 5.1 R2当前的动态性

| 层级 | R2是否具备 | 现状说明 |
|------|-----------|---------|
| 域层动态加载 | ❌ 没有 | 研究域和生产域是两个独立实例，不是动态加载关系 |
| 人格/角色动态切换 | ⚠️ 原始 | 靠system prompt切换，没有正式的路由机制 |
| 功能模块热插拔 | ❌ 没有 | 模块SOP是静态的，更新靠手动改文件 |
| 热更新校验机制 | ❌ 没有 | 更新全靠手动，没有自动校验和回滚 |

### 5.2 三个最值得借鉴的点

#### 1. 路由触发式激活，而非全量加载
现在R2的研究域什么都装了一整套系统，但大部分时间只用到其中一小部分。
可以借鉴的思路：
- 不是所有模块都常驻，而是**用到哪个模块被触发时才加载**
- 比如失效分析模块，只有真的出现失效时才激活，平时不占资源

#### 2. 人格=角色+职责+规则的三位一体
R1的每个人格都是"名字、职责、规则是绑定在一起的。
R2的lab_01和lab_02的分工也应该这样：
- ✅ 有名字（疯子/小疯子）
- ✅ 有职责（生产/研究）
- ⚠️ 规则还不够清晰明确（需要更明确的边界规则）

#### 3. 热更新的校验闭环
R1的人格激活有validation检查项，R2的模块更新也应该有：
- 更新前：备份当前版本快照
- 更新中：不中断当前任务
- 更新后：有明确的验收检查项
- 不通过：一键回滚

## 六、可立即落地的小改进

### 1. 给模块加"激活触发条件
每个模块SOP里加一个"什么时候激活"字段：
- M01信号验证：有新信号时激活
- M02失效分析：有失效事件时激活
- M03 Worker画像：有新Worker上线时激活
- M04 Constraint提案：有新的研究结论时激活
- M05同构检测：定期巡检时激活
- M06 R1考古：知识早班/午班时激活

不是所有模块都时刻"永远在线"，而是**被需要时才醒来干活。

### 2. 建立"更新=快照+校验"两步走
任何模块更新SOP时，遵循：
1. 先备份当前版本（快照）
2. 更新内容
3. 跑一遍检查清单（校验）
4. 校验不通过就回滚到快照

这和我们现在的交付前检查清单是配套的。

### 3. 明确双域的"热切换"边界
研究域和生产域之间，应该有明确的切换规则：
- 什么情况下生产域需要求助研究域？
- 研究域的结论什么时候可以同步到生产域？
- 同步过去之后怎么验证生效？

现在是靠人判断，以后可以做成更明确的触发规则。
