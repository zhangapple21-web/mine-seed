# M04 Constraint 提案模块

**模块ID**: M04
**职责**: 从研究证据中提炼可执行约束，分类归档，形成行为改变的闭环
**核心产出**: Constraint 草案 + 分类标签 + 生命周期管理
**状态**: v0.1 迭代中
**上游输入**: M01信号验证结论、M02失效分析报告、M03 Worker画像、R1考古发现
**下游输出**: 进入全局 Principle/Constraint 体系，指导未来研究和生产行为

---

## 核心思想

> 研究的终极目的不是产出报告，而是改变行为。
> 一条真正落地的Constraint，胜过十篇躺在文件夹里的分析报告。
> Constraint不是越多越好——每条都要算"维护成本"，太复杂的宁可不要。

---

## Constraint 分类体系

### A类：可程序化约束（硬约束）
- **定义**：可以写成代码、自动执行、可被机器校验的规则
- **特点**：精确、无歧义、执行成本低，但存活率低（~7%）
- **例子**："单只股票持仓不超过总仓位的5%"、"连续3天跌停强制止损"
- **适用场景**：边界清晰、对错分明、不需要人类判断的规则

### B类：软原则（弹性约束）
- **定义**：需要人理解和判断、不能完全自动化的指导原则
- **特点**：模糊但灵活、存活率高（~100%），但执行依赖判断力
- **例子**："不要追逐已经走出来的热点"、"优先考虑风险收益比，而非单纯的胜率"
- **适用场景**：复杂决策、需要上下文判断、没有绝对对错的情况

> **铁律**：A类约束宁少勿滥。写10条经常被突破的A类，不如1条坚守的B类。
> 存活率比精确性重要——一条活着的软原则，胜过十条死去的硬规则。

## Constraint 提案SOP（标准流程）

### Step 1: 触发源识别
什么事件触发了这条Constraint的提案？
- **失败触发**：M02发现了一个失效模式，需要避免再犯
- **成功触发**：M01验证了一个有效信号，需要固化为规则
- **观察触发**：从R1考古、外部资料、日常观察中发现值得借鉴的原则
- **需求触发**：生产环境提出了新的约束需求

### Step 2: 约束提炼
从原始素材中提炼出清晰的约束表述。

**好的Constraint标准**:
- ✅ **一句话能说清**：不要长篇大论
- ✅ **有明确的适用边界**：什么时候管用，什么时候不管用
- ✅ **可操作**：看完知道该怎么做/不该怎么做
- ✅ **可验证**：事后能判断有没有遵守
- ❌ 不是鸡汤："要理性"不是约束，是废话
- ❌ 不是目标："要赚钱"不是约束，是愿望

**约束表述公式**:
```
[在X条件下]，应该/不应该 [做Y]，因为 [Z原因]。
```

### Step 3: 分类判定
判断是A类还是B类。

| 判断维度 | A类（可程序化） | B类（软原则） |
|---------|---------------|-------------|
| 表述精度 | 精确、无歧义 | 模糊、需要解读 |
| 边界清晰度 | 边界明确 | 边界模糊，依赖上下文 |
| 执行方式 | 机器自动执行 | 人判断后执行 |
| 预期存活率 | ~7%（低） | ~100%（高） |
| 维护成本 | 低（一次写好） | 高（需要反复校准理解） |
| 犯错代价 | 高（错了就是硬错） | 低（有弹性空间） |

> **判断原则**：拿不准的，先归为B类。不要为了"显得高级"而硬写成A类。

### Step 4: 影响评估
评估这条Constraint如果落地，会有什么影响。
- **正面影响**：能避免什么错误？能提高什么效率？
- **负面影响**：会不会限制灵活性？会不会增加新的盲区？
- **适用范围**：适用于哪些场景？不适用于哪些场景？
- **冲突检查**：和已有的Constraint/Principle有没有冲突？

### Step 5: 生命周期设定
每条Constraint都有生命周期，不是写出来就永远有效。

| 阶段 | 说明 |
|------|------|
| **提案期** | 刚提出来，待验证 |
| **观察期** | 试运行中，收集证据 |
| **生效期** | 证据充分，正式生效 |
| **修订期** | 发现边界需要调整 |
| **废弃期** | 被证明无效或过时 |

**默认观察期**：A类观察3个月，B类观察6个月。

### Step 6: 提案输出
输出完整的Constraint提案，包括：
- 约束表述
- 分类（A/B）
- 触发源（从哪个研究/哪个事件来的）
- 适用边界
- 预期影响
- 生命周期阶段
- 验证计划（怎么验证它是不是真的有用）

## 二、背景与依据

### 2.2 证据支撑

### 2.3 为什么是现在

（什么时候应该遵守这条约束）

## 与其他模块的协作关系

- **从M01输入**：验证有效的信号，可以固化为A类约束或B类原则
- **从M02输入**：失效分析的结论，是Constraint提案最有价值的输入（避免再犯）
- **从M03输入**：Worker能力画像的边界，可以转化为"什么情况用什么Worker"的约束
- **从M06输入**：R1考古发现的架构原则，可以直接作为B类原则引入
- **向全局输出**：成熟的Constraint进入全局 Principle/Constraint 体系
- **向M05输出**：不同来源的Constraint可能表述不同但内涵一致，需要同构检测去重合并

## 维护原则

1. **有进有出**：定期review所有Constraint，过时的、没用的、被证伪的，及时废弃
2. **数量控制**：A类约束总数建议不超过20条，B类不超过50条。多了记不住，等于没有
3. **优先级排序**：按重要性排序，最重要的几条反复强调，比堆一堆没人看强
4. **版本管理**：每条Constraint的每次修订都留痕，知道为什么改、改了什么
5. **可证伪性**：每条Constraint都要有"什么情况下证明它是错的"的标准，不能永远正确
