# M03 Worker画像模块

**模块ID**: M03
**职责**: 绘制不同Worker的能力画像，明确适用场景、能力边界、成功率分布
**核心产出**: Worker能力矩阵、单Worker画像档案
**状态**: 已规划 · SOP完成 · 待积累数据

---

## 一、模块定位

Worker画像是研究域的"测量仪器校准模块"——我们不仅要知道"什么有效"，还要知道"谁来做最有效"。

每个Worker都有其能力边界和适用场景。画像的目的不是给Worker打分排名，而是建立**任务- Worker匹配度**的参考系，让分流决策有据可依。

> 核心原则：没有最好的Worker，只有最适合场景的Worker。

## 二、画像维度体系

### 2.1 基础属性层

| 维度 | 说明 | 数据类型 |
|------|------|----------|
| Worker ID | 唯一标识 | 字符串 |
| 显示名称 | 可读名称 | 字符串 |
| 类型 | 如：通用型/研究型/执行型/编码型 | 枚举 |
| 上线时间 | 首次部署时间 | 日期 |
| 当前版本 | 版本号 | 字符串 |
| 基础模型 | 底层依赖的大模型 | 字符串 |

### 2.2 能力标签层

从三个维度给Worker打标签：

- **任务域标签**：擅长的任务类型（如：信号分析、代码开发、文章写作、数据处理、创意生成）
- **风格标签**：输出风格特征（如：严谨型/发散型/简洁型/详尽型/保守型/激进型）
- **边界标签**：明确不擅长的领域（如：不擅长数学推导、不适合开放性创意）

### 2.3 性能指标层

| 指标 | 定义 | 统计周期 |
|------|------|----------|
| 完成率 | 成功交付的任务 / 总分配任务 | 近30天 |
| 返工率 | 需要返工的任务 / 成功交付任务 | 近30天 |
| 平均响应时长 | 从分配到交付的平均时间 | 近30天 |
| 准确率 | 输出无事实错误的比例 | 抽样统计 |
| 异常率 | 出现严重错误/幻觉/失控的比例 | 全量 |

按任务场景评级（S/A/B/C/D，S为最强）：

| 场景 | 评级 | 支撑样本数 | 备注 |

| 信号验证 | | | |
| 失效分析 | | | |
| 代码开发 | | | |
| 数据处理 | | | |
| 研究报告 | | | |
| 创意写作 | | | |
| 日常沟通 | | | |

### 2.5 失效模式层

记录该Worker常见的失效类型和触发条件（与M02联动）：

- 高频失效模式TOP3
- 典型触发条件
- 失效严重程度分布

### 2.6 置信度层

| 等级 | 判定标准 |
|------|----------|
| 高置信 | 样本量≥100，跨场景验证过 |
| 中置信 | 样本量30-100，主要场景有数据 |
| 低置信 | 样本量<30，以观察和推测为主 |

## 三、SOP 流程

### 3.1 新Worker画像建立（6步）

```
1. 基础信息录入 → 填好身份属性和初始标签
2. 初始观察期 → 分配3-5个不同类型任务，观察初步表现
3. 数据采集 → 记录每个任务的场景、耗时、结果质量、问题点
4. 初版画像 → 基于小样本生成第一版画像，标记"低置信"
5. 持续积累 → 随任务运行持续补充数据点
6. 置信度升级 → 样本量达标后升级置信度等级
```

### 3.2 画像更新机制

- **事件驱动更新**：每次M02分析发现某Worker的新失效模式时，同步更新画像
- **定期校准**：每月1号全量校准一次所有Worker的性能指标
- **版本迭代更新**：Worker版本升级后，重置部分指标，重新观察

### 3.3 数据采集方式

1. **生产域日志同步**：从lab_01的运行记录中自动抽取统计数据（待对接）
2. **研究任务记录**：研究域内部任务的执行表现
3. **专项测试**：设计标准化测试用例，批量评估Worker能力
4. **人工标注**：对输出质量做主观评级，补充客观指标覆盖不到的维度
