## 一、Capability 定义原则

### 1.1 定义原则

| 原则 | 说明 |
|------|------|
| **能力无关实现** | Capability 定义不依赖具体技术实现 |
| **语义精确** | 每个 Capability 有明确的语义边界 |
| **可组合** | Capability 可以组合成更复杂的能力 |
| **可验证** | Capability 的执行结果可验证 |
| **持久化友好** | Capability 的执行记录可持久化 |

### 1.2 Forbidden Rules

```
Forbidden

❌ 不要因为 Browser MCP 很新，就默认它是最佳方案。
❌ 不要因为 MCP 是标准，就认为 Civilization Repository 应该围绕 MCP 建立。
❌ 所有技术都必须回答三个问题：
   - 它解决了什么能力？
   - 这个能力是否已经存在？
   - 如果今天 Browser 消失，文明还能不能继续？
```

## 二、Capability 定义清单

### CAP-001：Reality Read

**ID**: CAP-001  
**名称**: Reality Read  
**语义**: 从现实世界读取信息  
**输入**: 读取目标描述、读取参数  
**输出**: 结构化/非结构化信息  
**验证方式**: 信息完整性校验  
**持久化**: 读取结果可持久化  

**能力边界**:
- 读取网页内容
- 读取 API 数据
- 读取文件系统
- 读取数据库
- 读取传感器数据

**R1 实现**:
- Selenium（网页读取）
- HTTP 客户端（API 读取）

**R2 实现**:
- KnowledgeService（知识库读取）
- 文件系统读取

---

## 三、Capability 关系图

```
┌─────────────────────────────────────────────────────────────────┐
│                    Capability Relationship                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  Reality Search ──► Reality Read ──► Reality Capture           │
│       │                   │                    │                │
│       │                   │                    ▼                │
│       ▼                   ▼            Reality Verify          │
│  Reality Observe ◄───────────────────────│                     │
│       │                                  │                     │
│       │                                  ▼                     │
│       ▼                           Reality Replay               │
│  Reality Write                                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 关系说明

1. **搜索 → 读取 → 捕获**: 先搜索目标，再读取内容，最后捕获证据
2. **读取 → 验证**: 读取的内容需要验证真实性
3. **观察 → 验证**: 观察到的变化需要验证
4. **验证 → 回放**: 验证过程可以回放
5. **写入 → 观察**: 写入操作后需要观察结果
6. **捕获 → 验证**: 捕获的证据用于验证

## 四、Capability 与 R2 公理对照

### 4.1 R2-1：结构 > 模型 ✅

Capability 定义是纯结构，不依赖任何模型或技术实现。

### 4.2 R2-2：统一身份 ✅

所有 Capability 执行都有统一的身份标识和记录。

### 4.3 R2-3：笨者生存 ✅

Capability 定义基于文件系统持久化，不依赖内存或网络。

### 4.4 R2-4：沉淀链 ✅

Capability 执行记录形成沉淀链：Observe → Evidence → Constraint → Route。

### 4.5 R2-5：连续性优先 ✅

Capability 定义保证了即使技术实现更换，能力本身保持连续。
