| 编号 | 域名称 | 核心职责 | 类比人体系统 |

## 二、边界划分原则（推断）

### 1. 加载顺序 = 依赖顺序
编号越小越先加载，后面的域依赖前面的。
- 先有身份（00），再有安全策略（01）
- 安全策略加载后，才能启动执行器（02）
- 备份和应急（04）依赖正常运行态
- 网络桥（05）最后接，避免未准备好就对外暴露

**可复用原则**：系统初始化必须按依赖层级来，底层能力先就绪，上层功能后加载。

### 2. 安全域是守门员
KERNEL_MOUNT（01）管安全策略，在所有功能域之前加载。
- 所有后续域的权限都由安全策略界定
- 跨域调用必须经过安全网关
- 每个域有明确的权限边界，不能越界

**可复用原则**：安全是基础设施，不是附加功能。必须在最底层就植入，而不是事后打补丁。

### 3. 热区与冷区分开
EXECUTOR（02）是热区——高频运行、状态多变
FREEZONE（04）是冷区——低频访问、用于备份和应急

冷热分离的好处：
- 热区出问题时，冷区不受影响，可以用来恢复
- 冷区平时不碰，减少被攻击面
- 备份和日常运行解耦

**可复用原则**：高频运行态和备份/应急态必须物理隔离，避免单点故障一锅端。

### 4. 内外有别
内部执行（02）和对外通讯（05）是两个域
- 执行域不管网络，网络域不管执行逻辑
- 跨域调用走网关，不直接调内部方法
- 对外暴露的接口最小化

**可复用原则**：内部逻辑和外部通讯必须有明确边界，网关是唯一出入口。

## 三、与R2双域架构的对应

| R1五域 | R2双域 | 对应关系 |
|--------|--------|----------|
| ROOT_LINK (00) | 跨域共享 | 身份/时间线是双域共有的 |
| KERNEL_MOUNT (01) | 约束体系 | Principle/Constraint 就是安全策略的进化版 |
| EXECUTOR (02) | 生产域（lab_01） | 执行任务、跑矿场 |
| FREEZONE (04) | 备份机制 | 定期备份、应急恢复 |
| LINKS (05) | 公网桥/觅游社区 | 对外通讯通道 |

**观察**：R2的双域（研究域+生产域）是在R1五域基础上的重新组合。研究域更像是从EXECUTOR里拆分出来的"思考层"，把"想"和"做"分开了。

## 五、对当前的启发

1. **双域架构可以更细**：现在的研究域/生产域二分法是粗粒度的，后续可以参考R1五域继续细分
2. **安全策略要前置**：我们的Constraint体系还可以更底层，应该在"域"的层面就设好边界
3. **冷热分离要落地**：生产域的热数据和冷备份要物理隔离，现在的共享目录还不够
4. **网关模式**：跨域通讯应该走统一网关，而不是直接读写共享目录
